9. Comments & Formatting

📚 Lesson 9 of 55 ⏱ 15 min read 🌱 Beginner

What Are Comments and Why Do They Matter?

Comments are lines of text in your source code that the compiler completely ignores. They exist solely for humans—you, your teammates, and your future self. Comments explain why the code does something, clarify tricky logic, and make large codebases navigable.

Good comments and clean formatting are hallmarks of professional software. Even if your code compiles and runs correctly, poor readability can make it nearly impossible to maintain, debug, or extend. In this lesson you will learn:

  • How to write single-line and multi-line comments
  • When comments help and when they hurt
  • Documentation comments (Doxygen style)
  • How to comment out code for debugging
  • Code formatting and style conventions
  • Naming conventions used in C++ projects
  • Header guards and file organization
  • Best practices for clean, readable code

Key Point

The compiler strips out every comment before it translates your code into machine instructions. Comments have zero effect on program size, speed, or behavior.

Single-Line Comments (//)

A single-line comment starts with two forward slashes (//). Everything from // to the end of that line is ignored by the compiler.

#include <iostream>
using namespace std;

int main() {
    // This is a single-line comment
    cout << "Hello, World!" << endl;  // Print greeting

    // You can stack multiple single-line comments
    // to form a paragraph of explanation.

    return 0;  // Exit successfully
}

Single-line comments are the most common type. Use them for brief notes next to a statement or short explanations above a block of code.

Inline vs. Above-Line Comments

Comments can appear at the end of a line (inline) or on their own line above the code they describe:

// Calculate the area of a circle (above-line comment)
double area = 3.14159 * radius * radius;

double volume = length * width * height;  // inline comment

Tip

Above-line comments are preferred for longer explanations, while inline comments work well for short clarifications. Keep inline comments brief so lines don’t become excessively long.

Multi-Line Comments (/* */)

A multi-line (block) comment begins with /* and ends with */. Everything between these delimiters is ignored, even if it spans many lines.

/*
   This program demonstrates
   multi-line comments in C++.
   The compiler ignores all of this text.
*/
#include <iostream>
using namespace std;

int main() {
    /* You can also use block comments on a single line */
    cout << "Block comments work!" << endl;
    return 0;
}

Key Point

Multi-line comments cannot be nested. If you write /* outer /* inner */ still inside? */, the compiler treats the first */ as the end of the comment, and the remaining text causes a compile error.

When to Use Multi-Line Comments

  • File-level descriptions at the top of a source file
  • Temporarily disabling a large block of code
  • License or copyright notices
  • Detailed algorithm explanations
/*
 * File: physics.cpp
 * Author: Jane Doe
 * Date: 2026-01-15
 * Description: Functions for basic physics calculations
 *              including velocity, acceleration, and force.
 */

#include "physics.h"

When and How to Write Good Comments

Not all comments are helpful. A bad comment can be worse than no comment at all if it is misleading or outdated. Follow these guidelines:

Comment the “Why,” Not the “What”

The code itself tells you what is happening. Comments should explain why a decision was made or why a non-obvious approach was chosen.

// BAD: States the obvious
int count = 0;  // Set count to zero

// GOOD: Explains WHY
int count = 0;  // Reset counter before each retry to avoid double-counting

Keep Comments Up to Date

When you change code, update the related comments. A comment that contradicts the code is extremely confusing.

// BAD: Comment says one thing, code does another
// Add 10% tax
double total = price * 1.15;  // Oops! This is actually 15%

// GOOD: Comment matches the code
// Add 15% sales tax
double total = price * 1.15;

Avoid Redundant Comments

// BAD: Every line has an obvious comment
int x = 5;        // Declare x as 5
x = x + 1;        // Add 1 to x
cout << x;        // Print x

// GOOD: Comment only where it adds value
int maxRetries = 5;  // Configured via ops team recommendation
maxRetries++;         // Account for the initial attempt
cout << maxRetries;

Tip

If you feel the need to write a long comment explaining a block of code, consider whether the code itself could be rewritten more clearly instead. Clear code with a descriptive function name often beats a comment.

Self-Documenting Code vs. Comments

Self-documenting code uses meaningful names and clear structure so that the code can be understood without comments. The goal is not to eliminate all comments, but to minimize the need for them.

Example: Poor Names Need Comments

// BAD: Cryptic names require comments
int d;  // elapsed time in days
double r;  // annual interest rate
bool f;  // true if file was found

Example: Good Names Speak for Themselves

// GOOD: No comments needed
int elapsedDays;
double annualInterestRate;
bool fileFound;

Extract Functions for Clarity

// BEFORE: Comment explains a complex block
// Check if the user is eligible for a discount
if (age >= 65 || isMember || totalPurchases > 1000.0) {
    applyDiscount(order);
}

// AFTER: Function name replaces the comment
if (isEligibleForDiscount(age, isMember, totalPurchases)) {
    applyDiscount(order);
}

Key Point

Self-documenting code reduces the maintenance burden because you only need to keep the code correct—there are no comments that can become stale. However, why-comments are still valuable even in well-written code.

Documentation Comments (Doxygen Style)

Documentation comments are specially formatted comments that tools like Doxygen can parse to generate HTML or PDF reference documentation automatically. C++ has two common Doxygen styles:

Javadoc Style: /** */

/**
 * @brief Calculates the factorial of a non-negative integer.
 *
 * Uses an iterative approach for efficiency.
 *
 * @param n The non-negative integer.
 * @return The factorial of n.
 * @throws std::invalid_argument if n is negative.
 */
long long factorial(int n) {
    if (n < 0) throw std::invalid_argument("Negative input");
    long long result = 1;
    for (int i = 2; i <= n; ++i) {
        result *= i;
    }
    return result;
}

Triple-Slash Style: ///

/// @brief Finds the maximum value in an array.
/// @param arr Pointer to the first element.
/// @param size Number of elements in the array.
/// @return The maximum value found.
int findMax(const int* arr, int size) {
    int maxVal = arr[0];
    for (int i = 1; i < size; ++i) {
        if (arr[i] > maxVal) maxVal = arr[i];
    }
    return maxVal;
}

Common Doxygen Tags

TagPurpose
@briefShort summary of the function
@paramDescribes a function parameter
@returnDescribes the return value
@throwsDocuments exceptions the function may throw
@seeCross-reference to related functions or classes
@noteAdditional notes or caveats
@deprecatedMarks a function as deprecated
@fileDescribes the file
@authorLists the author
@dateCreation or modification date
 
/**
 * @file math_utils.h
 * @author John Smith
 * @date 2026-02-01
 * @brief Utility functions for common math operations.
 */

#ifndef MATH_UTILS_H
#define MATH_UTILS_H

/**
 * @brief Computes the greatest common divisor of two integers.
 * @param a First integer.
 * @param b Second integer.
 * @return The GCD of a and b.
 * @note Uses the Euclidean algorithm.
 * @see lcm()
 */
int gcd(int a, int b);

/**
 * @brief Computes the least common multiple of two integers.
 * @param a First integer.
 * @param b Second integer.
 * @return The LCM of a and b.
 */
int lcm(int a, int b);

#endif // MATH_UTILS_H

Tip

Most modern IDEs (Visual Studio, CLion, VS Code with extensions) can parse Doxygen comments and show the documentation in tooltips when you hover over a function call. This makes Doxygen comments doubly useful.

Commenting Out Code for Debugging

One practical use of comments is temporarily disabling code while debugging. This lets you isolate a problem without deleting code you may want to restore later.

Using Single-Line Comments

int main() {
    int result = compute();
    // cout << "Debug: result = " << result << endl;
    process(result);
    return 0;
}

Using Multi-Line Comments

int main() {
    int result = compute();

    /*
    cout << "Step 1: " << result << endl;
    result = transform(result);
    cout << "Step 2: " << result << endl;
    */

    process(result);
    return 0;
}

Using #if 0 (Preprocessor Trick)

Because multi-line comments cannot nest, a common C++ idiom is to use the preprocessor to disable blocks of code:

#if 0
    // This entire block is excluded from compilation.
    // Even nested /* block comments */ are safe here.
    cout << "This will NOT be compiled." << endl;
#endif

Key Point

Commented-out code should be temporary. If code is no longer needed, delete it—version control (Git) will preserve the history. Dead code clutters the codebase and confuses other developers.

Code Formatting and Style Conventions

Consistent formatting makes code easier to read, review, and maintain. While C++ does not enforce any particular style, teams typically agree on a style guide and use automated formatters to enforce it.

Why Formatting Matters

  • Readability – Well-formatted code is easier to scan and understand.
  • Consistency – A uniform style across a project reduces cognitive load.
  • Code reviews – Consistent formatting lets reviewers focus on logic, not style.
  • Bug prevention – Proper indentation makes control flow obvious and avoids mistakes like dangling else.

Indentation: Tabs vs. Spaces

Indentation visually shows the structure and nesting of your code. The two main approaches are:

StyleProsCons
Spaces (2 or 4)Looks identical in every editor; most style guides recommend itSlightly larger file size (negligible)
TabsOne character per indent; user-configurable widthDisplays differently in various editors; can misalign if mixed with spaces
 
// 4-space indentation (most common in C++)
int main() {
    if (true) {
        for (int i = 0; i < 10; ++i) {
            cout << i << endl;
        }
    }
    return 0;
}

Tip

The Google C++ Style Guide and the LLVM style both use spaces. Many editors let you press Tab and insert spaces automatically (“soft tabs”). The most important rule: never mix tabs and spaces in the same file.

Brace Placement Styles

Two popular brace placement styles in C++:

// K&R / "One True Brace Style" (1TBS)
// Opening brace on the same line
if (condition) {
    doSomething();
} else {
    doOther();
}

// Allman Style
// Opening brace on its own line
if (condition)
{
    doSomething();
}
else
{
    doOther();
}

Neither style is objectively better. Choose one and apply it consistently throughout your project.

Naming Conventions

Consistent naming is one of the most impactful ways to improve code readability. C++ does not enforce naming rules, but the community has well-established conventions:

ConventionExampleTypical Use
camelCaseplayerScoreLocal variables, function parameters
PascalCaseGameEngineClass names, struct names, type aliases
snake_casetotal_countVariables, functions (STL style)
UPPER_CASEMAX_SIZEConstants, macros, enum values
m_prefix or trailing _m_name / name_Member variables (to distinguish from locals)
 
#include <iostream>
#include <string>
using namespace std;

const int MAX_STUDENTS = 30;       // UPPER_CASE for constants

class StudentRecord {               // PascalCase for classes
public:
    void setName(const string& newName) {  // camelCase for functions
        m_name = newName;            // m_ prefix for member variables
    }

    string getName() const {
        return m_name;
    }

private:
    string m_name;                   // member variable
    int m_age = 0;
};

int main() {
    int studentCount = 0;            // camelCase for local variables
    StudentRecord record;
    record.setName("Alice");
    cout << record.getName() << endl;
    return 0;
}

Key Point

The C++ Standard Library uses snake_case for everything (e.g., std::vector, push_back, size_type). Many teams follow the STL convention, while others prefer camelCase or PascalCase for their own code. What matters most is consistency.

Naming Tips

  • Use descriptive names: calculateTax() is better than calc().
  • Avoid abbreviations unless universally understood (max, min, idx).
  • Boolean variables should read like a question: isReady, hasPermission, canProceed.
  • Loop counters like i, j, k are acceptable for short loops.
  • Avoid names that differ only by case: data vs. Data is confusing.
  • Never start identifiers with an underscore followed by an uppercase letter (_Name)—those are reserved by the C++ standard.

Whitespace and Line Length Guidelines

Strategic use of blank lines and spaces improves readability dramatically.

Horizontal Whitespace

// BAD: No spaces
int x=a+b*c;
if(x>0){cout<<x;}

// GOOD: Spaces around operators and after keywords
int x = a + b * c;
if (x > 0) {
    cout << x;
}

Vertical Whitespace (Blank Lines)

#include <iostream>
#include <string>
                                    // Blank line after includes
using namespace std;
                                    // Blank line before function
void greet(const string& name) {
    cout << "Hello, " << name << "!" << endl;
}
                                    // Blank line between functions
int main() {
    greet("World");
    return 0;
}

Line Length

Most style guides recommend limiting lines to 80–120 characters. Long lines are hard to read and cause horizontal scrolling. Break long statements logically:

// BAD: Excessively long line
double result = calculateCompoundInterest(principal, annualRate, timesCompoundedPerYear, numberOfYears);

// GOOD: Break after the opening parenthesis or at commas
double result = calculateCompoundInterest(
    principal,
    annualRate,
    timesCompoundedPerYear,
    numberOfYears
);

Tip

Tools like clang-format can automatically reformat your code according to a style configuration file (.clang-format). This eliminates formatting debates and ensures consistency across a team.

Header Guards and File Organization

When you split code across multiple files (headers and source files), header guards prevent the same header from being included more than once, which would cause redefinition errors.

Traditional Header Guards (#ifndef)

// File: player.h
#ifndef PLAYER_H
#define PLAYER_H

#include <string>

class Player {
public:
    Player(const std::string& name, int health);
    std::string getName() const;
    int getHealth() const;

private:
    std::string m_name;
    int m_health;
};

#endif // PLAYER_H

#pragma once (Modern Alternative)

// File: player.h
#pragma once

#include <string>

class Player {
public:
    Player(const std::string& name, int health);
    std::string getName() const;
    int getHealth() const;

private:
    std::string m_name;
    int m_health;
};

#pragma once is not part of the C++ standard but is supported by all major compilers (GCC, Clang, MSVC). Many modern projects prefer it for its simplicity.

Recommended File Organization

// Recommended order inside a .cpp file:

// 1. Matching header (e.g., "player.cpp" includes "player.h" first)
#include "player.h"

// 2. C system headers
#include <cstdlib>
#include <cmath>

// 3. C++ standard library headers
#include <iostream>
#include <string>
#include <vector>

// 4. Other library headers
#include <nlohmann/json.hpp>

// 5. Project headers
#include "game_engine.h"
#include "utils.h"

// 6. Namespaces, constants, and code
using namespace std;

const int MAX_PLAYERS = 4;

// ... function implementations ...

Key Point

Placing the matching header first (e.g., #include "player.h" in player.cpp) ensures the header is self-contained—if it’s missing any includes, the compiler will catch the error immediately.

Best Practices for Clean, Readable Code

1. One Statement per Line

// BAD: Multiple statements on one line
int a = 1; int b = 2; int c = a + b;

// GOOD: One statement per line
int a = 1;
int b = 2;
int c = a + b;

2. Declare Variables Close to First Use

// BAD: All declarations at the top (C89 style)
int i, j, sum, count;
// ... 50 lines later ...
sum = 0;
for (i = 0; i < count; ++i) { /* ... */ }

// GOOD: Declare where needed
int sum = 0;
for (int i = 0; i < count; ++i) {
    sum += data[i];
}

3. Use const Whenever Possible

const double PI = 3.14159265358979;
const int MAX_RETRIES = 3;

void printMessage(const string& msg) {
    cout << msg << endl;
    // msg cannot be modified here
}

4. Keep Functions Short and Focused

// BAD: One massive function that does everything
void processOrder(Order& order) {
    // 200 lines of validation, calculation, logging, emailing...
}

// GOOD: Break into smaller, well-named functions
void validateOrder(const Order& order);
double calculateTotal(const Order& order);
void logTransaction(const Order& order);
void sendConfirmation(const Order& order);

void processOrder(Order& order) {
    validateOrder(order);
    double total = calculateTotal(order);
    order.setTotal(total);
    logTransaction(order);
    sendConfirmation(order);
}

5. Avoid Magic Numbers

// BAD: What does 86400 mean?
int seconds = days * 86400;

// GOOD: Named constant explains the meaning
const int SECONDS_PER_DAY = 86400;
int seconds = days * SECONDS_PER_DAY;

6. Use a Consistent Style Guide

Popular C++ style guides you can adopt or adapt:

  • Google C++ Style Guide – Widely used in industry; opinionated but thorough.
  • C++ Core Guidelines – Authored by Bjarne Stroustrup and Herb Sutter; focuses on modern C++.
  • LLVM Coding Standards – Used by the LLVM/Clang project.
  • Mozilla C++ Style Guide – Used in the Firefox codebase.

Tip

Using an automated formatter like clang-format eliminates style debates. Create a .clang-format file at the root of your project and let the tool handle indentation, brace placement, spacing, and line wrapping automatically.

Summary

  • Single-line comments (//) are for brief notes; most commonly used in C++.
  • Multi-line comments (/* */) span multiple lines but cannot be nested.
  • Write comments that explain why, not what; keep them accurate and up to date.
  • Self-documenting code with clear names reduces the need for comments.
  • Documentation comments (/** */, ///) feed tools like Doxygen to generate reference docs.
  • Use comments or #if 0 to temporarily disable code while debugging.
  • Consistent indentation (prefer spaces), brace style, and line length improve readability.
  • Follow naming conventions: PascalCase for types, camelCase or snake_case for variables/functions, UPPER_CASE for constants.
  • Use header guards (#ifndef or #pragma once) to prevent double inclusion.
  • Organize files logically and use automated formatters like clang-format.
