53. Version Control with Git

📚 Lesson 53 of 55 ⏱ 16 min read 🔨 Advanced

What Is Version Control?

Version control (also called source control or revision control) is a system that records changes to files over time so you can recall specific versions later. For C++ developers working on projects that may contain hundreds of source and header files, version control is not optional – it is essential.

Without version control, teams resort to copying folders with names like project_v2_final_FINAL – an error-prone approach that quickly becomes unmanageable. A proper version control system solves this by maintaining a complete, searchable history of every change.

Key benefits of version control:

  • History – Every change is recorded with a timestamp, author, and description.
  • Collaboration – Multiple developers can work on the same codebase simultaneously without overwriting each other’s work.
  • Branching – You can experiment with new features in isolation and merge them when ready.
  • Recovery – If something breaks, you can revert to any previous working state.
  • Accountability – You can see exactly who changed what and why.

There are two main categories of version control systems:

  • Centralised (CVCS) – A single server holds the repository history (e.g., SVN, Perforce). Developers check out files from the server, make changes, and commit back. If the server goes down, no one can commit.
  • Distributed (DVCS) – Every developer clones the full repository, including its entire history (e.g., Git, Mercurial). You can commit, branch, and view history entirely offline. The server is used only for sharing changes.

Git is the most widely used version control system in the world. Created by Linus Torvalds in 2005 to manage the Linux kernel, Git is distributed, meaning every developer has a full copy of the repository history on their local machine. This makes it fast, resilient, and suitable for projects of any size – from a single-file homework assignment to a million-line enterprise codebase.

Git’s design goals were speed, data integrity, and support for non-linear workflows (thousands of parallel branches). These qualities make it the de facto standard for both open-source and commercial software development.

Installing Git

Git is available on all major operating systems. After installation, verify it by running git --version in your terminal.

Windows

Download the official installer from git-scm.com and run it. The installer includes Git Bash, a Unix-like terminal that provides a consistent command-line experience on Windows.

Terminal (Windows)
# Verify installation
git --version
# git version 2.44.0.windows.1

macOS

Git ships with the Xcode Command Line Tools. Install them with a single command:

Terminal (macOS)
xcode-select --install
git --version

Alternatively, install via Homebrew: brew install git.

Linux (Debian/Ubuntu)

Terminal (Linux)
sudo apt update && sudo apt install git -y
git --version

Initial Configuration

After installing Git, set your identity. This information is embedded in every commit you make.

Terminal
git config --global user.name "Your Name"
git config --global user.email "you@example.com"
git config --global core.editor "code --wait"   # Use VS Code as editor
git config --global init.defaultBranch "main"    # Default branch name

View all your settings with git config --list.

Git Basics

A repository (repo) is a directory whose history is tracked by Git. Git stores its metadata in a hidden .git/ folder at the root of the repository.

Creating a Repository

Terminal
# Create a new project and initialise a Git repo
mkdir my_cpp_project
cd my_cpp_project
git init
# Initialized empty Git repository in /home/user/my_cpp_project/.git/

Cloning an Existing Repository

Terminal
# Clone a remote repository to your machine
git clone https://github.com/username/project.git
cd project

The Three Areas

Git uses three main areas to manage your files:

  1. Working Directory – The actual files on your disk that you edit.
  2. Staging Area (Index) – A holding area where you prepare changes before committing.
  3. Repository (.git) – The permanent history of all committed snapshots.

Basic Workflow: status → add → commit

Terminal
# Check which files have changed
git status

# Stage specific files
git add main.cpp utils.h

# Stage all changed files
git add .

# Commit with a descriptive message
git commit -m "Add utility functions for matrix operations"

Each commit is a snapshot of your entire project at a point in time. Commits are identified by a 40-character SHA-1 hash (e.g., a1b2c3d...).

Unstaging and Discarding Changes

Terminal
# Unstage a file (keep changes in working directory)
git restore --staged main.cpp

# Discard changes in working directory (revert to last commit)
git restore main.cpp

# Amend the most recent commit (e.g., fix the message or add a file)
git add forgotten_file.cpp
git commit --amend -m "Add matrix utilities and forgotten helper"

Warning: git restore permanently discards uncommitted changes. Always double-check with git diff before restoring.

Viewing History

Git provides several commands to inspect the history of your project.

git log

Terminal
# Full log
git log

# One-line summary per commit
git log --oneline

# Graphical branch view
git log --oneline --graph --all

# Show commits by a specific author
git log --author="Alice"

# Show commits that modified a specific file
git log -- src/main.cpp

git diff

Terminal
# See unstaged changes (working dir vs staging)
git diff

# See staged changes (staging vs last commit)
git diff --staged

# Compare two commits
git diff abc1234 def5678

# Compare two branches
git diff main feature-branch

git show

Terminal
# Show details and diff of a specific commit
git show abc1234

# Show a file at a specific commit
git show abc1234:src/main.cpp

Branching

Branches are one of Git’s most powerful features. A branch is simply a lightweight, movable pointer to a commit. Branches let you work on features, bug fixes, or experiments in isolation without affecting the main branch.

Terminal
# List all branches (* marks the current branch)
git branch

# Create a new branch
git branch feature-parser

# Switch to the new branch
git checkout feature-parser
# or (Git 2.23+)
git switch feature-parser

# Create and switch in one command
git checkout -b feature-parser
# or
git switch -c feature-parser

# Delete a branch that has been merged
git branch -d feature-parser

# Force-delete an unmerged branch
git branch -D feature-parser

A good branching strategy keeps main stable and deployable at all times while development happens on short-lived feature branches.

Listing Remote Branches

Terminal
# List all branches including remote-tracking branches
git branch -a

# Fetch and list newly created remote branches
git fetch --all
git branch -r

Renaming a Branch

Terminal
# Rename the current branch
git branch -m new-name

# Rename a different branch
git branch -m old-name new-name

Merging and Resolving Conflicts

After finishing work on a branch, you merge it back into main (or another target branch). Git tries to combine the changes automatically.

Fast-Forward Merge

If main has not diverged since the branch was created, Git simply moves the main pointer forward – no merge commit is needed.

Terminal
git switch main
git merge feature-parser
# Fast-forward merge – no conflicts possible

Three-Way Merge

If both branches have new commits, Git performs a three-way merge and creates a merge commit.

Handling Merge Conflicts

A conflict occurs when both branches modify the same lines. Git marks the conflicting sections in the file:

main.cpp (conflict markers)
<<<<<<< HEAD
int bufferSize = 1024;
=======
int bufferSize = 4096;
>>>>>>> feature-parser

To resolve the conflict:

  1. Open the file and decide which version to keep (or combine both).
  2. Remove the conflict markers (<<<<<<<, =======, >>>>>>>).
  3. Stage the resolved file and commit:
Terminal
git add main.cpp
git commit -m "Resolve buffer size conflict: use 4096"

Tip: Use git mergetool to launch a visual merge tool (e.g., VS Code, Meld, or KDiff3) for complex conflicts.

Remote Repositories

A remote is a version of your repository hosted on a server (e.g., GitHub, GitLab, Bitbucket). Remotes enable collaboration by providing a shared source of truth.

Terminal
# List configured remotes
git remote -v

# Add a remote (typically called "origin")
git remote add origin https://github.com/username/project.git

# Push your main branch to the remote
git push -u origin main

# Push a feature branch
git push origin feature-parser

# Fetch changes from the remote (does NOT modify your files)
git fetch origin

# Pull = fetch + merge into current branch
git pull origin main

The -u flag in git push -u origin main sets the upstream tracking, so subsequent pushes only need git push.

GitHub Workflow

The GitHub Flow is a lightweight, branch-based workflow popular in open-source and professional projects:

  1. Fork the repository on GitHub (for open-source contributions).
  2. Clone your fork locally: git clone https://github.com/you/project.git
  3. Create a branch for your change: git switch -c fix-memory-leak
  4. Make commits with clear messages.
  5. Push your branch: git push origin fix-memory-leak
  6. Open a Pull Request (PR) on GitHub, describing the change.
  7. Collaborators review the code, suggest changes, and approve.
  8. Merge the PR into main once approved.
  9. Delete the feature branch after merging.

Pull requests are more than just merge mechanisms – they serve as a discussion forum where team members review code, run automated tests (CI/CD), and ensure quality before changes reach the main branch.

Keeping Your Fork Up to Date

Terminal
# Add the original repo as "upstream"
git remote add upstream https://github.com/original/project.git

# Fetch upstream changes
git fetch upstream

# Merge upstream/main into your local main
git switch main
git merge upstream/main

# Push updated main to your fork
git push origin main

.gitignore for C++ Projects

A .gitignore file tells Git which files and directories to ignore. For C++ projects, you should ignore build artefacts, compiled binaries, IDE-specific files, and operating-system generated files.

out/

Tip: If you accidentally committed a file before adding it to .gitignore, remove it from tracking with:

Terminal
git rm --cached build/output.exe
git commit -m "Stop tracking build output"

Git Best Practices

Writing Good Commit Messages

A well-written commit message makes the project history easy to navigate. Follow the widely adopted conventional format:

Commit message format
# Subject line: imperative mood, ≤ 50 chars
Fix buffer overflow in MatrixParser::read()

# Body (optional): wrapped at 72 chars, explains WHY
The parser did not validate column count before indexing
into the data array, causing an out-of-bounds write when
the input file contained ragged rows.

Closes #142

Rules of thumb:

  • Use the imperative mood: “Fix bug” not “Fixed bug” or “Fixes bug.”
  • Keep the subject line under 50 characters.
  • Separate subject from body with a blank line.
  • Wrap body text at 72 characters.
  • Explain why the change was made, not just what changed.
  • Reference issue numbers (e.g., Closes #142) when applicable.

Branching Strategy

For most C++ projects, a simple strategy works best:

  • main – Always contains stable, tested code.
  • feature/* – Short-lived branches for new features (e.g., feature/json-parser).
  • bugfix/* – Branches for bug fixes (e.g., bugfix/null-ptr-crash).
  • release/* – Preparation branches for upcoming releases (e.g., release/2.0).

General Best Practices

  • Commit early and often – Small, focused commits are easier to review and revert.
  • Never commit broken code to main – Use feature branches.
  • Pull before you push – Always git pull before pushing to avoid conflicts.
  • Use .gitignore from day one – Don’t let build artefacts pollute the repo.
  • Tag releases – Use git tag -a v1.0 -m "Release 1.0" to mark stable versions.
  • Review diffs before committing – Run git diff --staged to verify what you are about to commit.
  • Learn to use git stash – Temporarily shelve uncommitted changes when you need to switch branches.

Key Takeaways

  • Version control records every change to your codebase, enabling history tracking, collaboration, branching, and recovery.
  • Git is a distributed version control system – every clone is a full backup of the repository.
  • The core workflow is status → add → commit. Keep commits small, focused, and well-described.
  • Branches let you develop features in isolation. Merge them into main when complete and tested.
  • Conflicts happen when two branches modify the same code. Resolve them by editing the conflict markers and committing.
  • Remotes (e.g., GitHub) provide a shared hub for collaboration. Use push, pull, and fetch to synchronise.
  • Use a .gitignore to keep build artefacts, binaries, and IDE files out of the repository.
  • Write imperative, concise commit messages that explain why a change was made.
  • Adopt a branching strategy (e.g., main + feature branches) to keep the project organised.
  • The Pull Request workflow combines code review, automated testing, and discussion into a single, traceable process.
