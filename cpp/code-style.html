52. Code Style & Documentation

📚 Lesson 52 of 55 ⏱ 14 min read 🔨 Advanced

Why Code Style Matters

Code style is the set of conventions that govern how source code is written and formatted. While the compiler does not care whether you use tabs or spaces, humans read code far more often than they write it. Consistent style dramatically improves readability, maintainability, and team collaboration.

Studies show that developers spend roughly 70–80% of their time reading existing code rather than writing new code. A well-styled codebase reduces cognitive load, makes bugs easier to spot, and accelerates onboarding for new team members.

Key benefits of a consistent code style:

  • Readability – Anyone can quickly understand the intent of the code.
  • Maintainability – Modifications are safer and faster when the codebase is uniform.
  • Collaboration – Teams avoid “style wars” during code reviews.
  • Fewer bugs – Clean, well-organised code makes errors stand out.
  • Automation – Tools like clang-format can enforce style automatically.

The golden rule: pick a style guide and follow it consistently. The specific style matters far less than consistency across the entire project.

Naming Conventions

Good names are one of the most important aspects of readable code. A well-chosen name tells the reader what something is and why it exists without requiring a comment.

Variables & Functions

Use descriptive names that convey meaning. Avoid single-letter names except for loop counters or well-known mathematical variables.

// Bad
int x = 42;
void fn(int a, int b);

// Good – snake_case (common in STL-like code)
int student_count = 42;
void calculate_average(int total_score, int num_students);

// Good – camelCase (common in many C++ projects)
int studentCount = 42;
void calculateAverage(int totalScore, int numStudents);

Classes & Structs

Use PascalCase (also called UpperCamelCase) for class and struct names. This is universally adopted across almost every C++ style guide.

class HttpRequest { };
struct PixelColor { };
class FileManager { };

Constants & Enumerators

Constants and enum values are typically written in ALL_CAPS with underscores, or with a k prefix in PascalCase (Google style).

// ALL_CAPS style
constexpr int MAX_BUFFER_SIZE = 1024;
enum class Colour { RED, GREEN, BLUE };

// Google k-prefix style
constexpr int kMaxBufferSize = 1024;

Files

Header files use .h or .hpp; source files use .cpp. File names should match the primary class they contain and use snake_case or kebab-case:

http_request.h    // header
http_request.cpp  // source
// or
HttpRequest.h
HttpRequest.cpp

Prefixes & Suffixes

Some teams use a prefix or suffix to distinguish member variables:

class Widget {
    int m_width;   // m_ prefix for members
    int width_;    // trailing underscore (Google style)
};

Indentation and Formatting

Consistent indentation is the foundation of readable code. The two dominant schools are spaces and tabs. Most modern C++ style guides prefer 4 spaces per indent level (Google uses 2 spaces).

  • Spaces – Render identically on every editor and terminal.
  • Tabs – Allow each developer to choose their visual width.

Configure your editor to insert spaces when you press the Tab key. This prevents mixing tabs and spaces, which creates misaligned code in different environments.

// 4-space indentation
if (isValid) {
    processData();
    if (hasMore) {
        loadNextBatch();
    }
}

Brace Styles

Where you place curly braces is one of the most debated formatting choices. Here are the three most common styles used in C++:

K&R (Kernighan & Ritchie) Style

The opening brace goes on the same line as the statement. This is the most common style in modern C++ projects and is used by the Google and LLVM style guides.

if (condition) {
    doSomething();
} else {
    doOther();
}

void process() {
    // ...
}

Allman Style

The opening brace goes on its own line, aligned with the statement. This style is popular in C# and some C++ codebases.

if (condition)
{
    doSomething();
}
else
{
    doOther();
}

void process()
{
    // ...
}

Stroustrup Style

A variant of K&R where else, catch, etc. start on a new line after the closing brace. Named after Bjarne Stroustrup, the creator of C++.

if (condition) {
    doSomething();
}
else {
    doOther();
}

void process() {
    // ...
}

All three styles are valid. The most important rule is: never mix styles within a project.

Line Length and Whitespace

Most style guides recommend a maximum line length of 80 or 120 characters. Long lines force horizontal scrolling and make side-by-side diffs harder to read.

Strategies for handling long lines:

// Break after an operator
double result = firstOperand + secondOperand
              + thirdOperand + fourthOperand;

// Break function arguments
void sendMessage(const std::string& recipient,
                 const std::string& subject,
                 const std::string& body,
                 Priority priority);

Use blank lines to separate logical sections within a function, but avoid excessive blank lines:

void processOrder(const Order& order) {
    // Validate
    if (!order.isValid()) {
        throw std::invalid_argument("Invalid order");
    }

    // Calculate totals
    double subtotal = calculateSubtotal(order);
    double tax = subtotal * TAX_RATE;
    double total = subtotal + tax;

    // Persist
    database.save(order, total);
}

Use spaces around binary operators and after commas, but not inside parentheses:

// Good
int sum = a + b;
call(x, y, z);

// Bad
int sum=a+b;
call( x,y,z );

Commenting Best Practices

Comments should explain why, not what. The code itself should be clear enough to convey what it does. Reserve comments for intent, rationale, warnings, and non-obvious behaviour.

When to Comment

  • Why a decision was made (e.g., “Using insertion sort because n < 16 in practice”)
  • Workarounds for bugs or platform quirks
  • TODOs and FIXMEs for future work
  • Legal notices and copyright headers
  • Public API documentation (see Doxygen below)

When NOT to Comment

  • Restating what the code already says: i++; // increment i
  • Commenting out dead code – delete it (version control remembers)
  • Describing obvious variable names: int age; // the age

Comment Styles

// Single-line comment – preferred for brief notes

/*
 * Multi-line comment – good for longer explanations,
 * file headers, and legal notices.
 */

// TODO(alice): Refactor this after the API migration
// FIXME: Off-by-one error when index == max_size

Documentation Comments (Doxygen Format)

Doxygen is the de-facto standard tool for generating API documentation from annotated C++ source code. It reads specially-formatted comments and produces HTML, PDF, or man-page output.

Basic Doxygen Syntax

/**
 * @brief Calculates the area of a circle.
 *
 * Uses the formula A = πr². The radius must be
 * non-negative; otherwise the behaviour is undefined.
 *
 * @param radius  The radius of the circle (must be ≥ 0).
 * @return The area of the circle.
 *
 * @throws std::invalid_argument if radius is negative.
 *
 * @see circumference()
 */
double area(double radius);

Common Doxygen Tags

TagPurpose
@briefShort one-line summary
@paramDescribes a function parameter
@returnDescribes the return value
@throwsDocuments exceptions thrown
@seeCross-reference another symbol
@noteHighlights important information
@warningWarns about potential pitfalls
@deprecatedMarks a symbol as deprecated
@fileDocuments the current file
@classDocuments a class
@code / @endcodeEmbeds a code example

Documenting a Class

/**
 * @class Matrix
 * @brief A simple 2D matrix of doubles.
 *
 * Supports basic arithmetic operations and element access.
 * Stores elements in row-major order.
 *
 * @tparam T The element type.
 *
 * @code
 * Matrix<double> m(3, 3);
 * m(0, 0) = 1.0;
 * @endcode
 */
template <typename T>
class Matrix { /* ... */ };

Generating Documentation

Install Doxygen, then create a configuration file and build:

# Generate a default config file
doxygen -g Doxyfile

# Edit Doxyfile to set PROJECT_NAME, INPUT, etc.

# Generate documentation
doxygen Doxyfile

The output (typically HTML) can be hosted on a web server or browsed locally.

Header Guards and #pragma once

Header files must be protected against multiple inclusion. Without protection, the same declarations appear multiple times, causing compilation errors.

Traditional Header Guards

#ifndef HTTP_REQUEST_H
#define HTTP_REQUEST_H

class HttpRequest {
    // ...
};

#endif // HTTP_REQUEST_H

The macro name should be unique – typically the file path in uppercase with underscores replacing slashes and dots.

#pragma once

#pragma once

class HttpRequest {
    // ...
};

#pragma once is simpler, avoids macro name collisions, and is supported by all major compilers (GCC, Clang, MSVC). However, it is not part of the ISO C++ standard, so some strict codebases still prefer traditional guards.

Code Organisation

Properly organising your source files is essential for scaling beyond a single-file program. The standard C++ approach separates declarations (in headers) from definitions (in source files).

Header Files (.h / .hpp)

  • Class declarations
  • Function prototypes
  • inline and constexpr function definitions
  • Template definitions (must be in headers)
  • Type aliases and constants

Source Files (.cpp)

  • Function and method definitions
  • Static variable initialisation
  • Implementation details hidden from consumers

Example Structure

// vector3d.h
#pragma once

class Vector3D {
public:
    Vector3D(double x, double y, double z);
    double magnitude() const;
    Vector3D normalised() const;

private:
    double x_, y_, z_;
};

// vector3d.cpp
#include "vector3d.h"
#include <cmath>

Vector3D::Vector3D(double x, double y, double z)
    : x_(x), y_(y), z_(z) {}

double Vector3D::magnitude() const {
    return std::sqrt(x_ * x_ + y_ * y_ + z_ * z_);
}

Vector3D Vector3D::normalised() const {
    double mag = magnitude();
    return {x_ / mag, y_ / mag, z_ / mag};
}

Forward Declarations

Use forward declarations instead of #include when only a pointer or reference to a type is needed. This reduces compilation time and breaks circular dependencies:

// Avoid: #include "database.h"
class Database; // forward declaration

class UserService {
public:
    void save(const Database& db);
};

Include Order

A recommended include order (Google style) to catch missing headers early:

  1. The corresponding header (e.g., foo.h in foo.cpp)
  2. C system headers (<cstdlib>)
  3. C++ standard library headers (<vector>)
  4. Third-party library headers
  5. Your project’s headers
// foo.cpp
#include "foo.h"

#include <cstdlib>

#include <algorithm>
#include <string>
#include <vector>

#include <fmt/core.h>

#include "bar.h"
#include "utils/logger.h"

Popular C++ Style Guides

Rather than inventing your own conventions from scratch, adopt one of the widely-used style guides listed below. Each has extensive documentation and tooling support.

Style GuideKey Characteristics
Google C++ Style Guide 2-space indent, 80-char lines, snake_case for variables/functions, PascalCase for types, k-prefix constants, trailing underscores for members. Very detailed and prescriptive.
LLVM Coding Standards 2-space indent, 80-char lines, PascalCase for nearly everything, K&R braces. Used by the LLVM/Clang project.
Mozilla C++ Style 2-space indent, 80-char lines, camelCase for functions, PascalCase for types, m-prefix for members. Used by Firefox.
C++ Core Guidelines Authored by Bjarne Stroustrup and Herb Sutter. Focuses on safety, performance, and modern idioms rather than cosmetic formatting.
Microsoft C++ Guidelines 4-space indent, Allman braces, PascalCase functions, widely used in Windows/MSVC projects.

Linting Tools

Linting tools automatically check and enforce code style, detect potential bugs, and suggest improvements. Integrating them into your build pipeline ensures consistent quality.

clang-format

clang-format is the industry-standard tool for automatically formatting C++ code. It supports all major style guides out of the box.

# Format a single file using the Google style
clang-format -i --style=Google main.cpp

# Format all .cpp and .h files recursively
find src/ -name "*.cpp" -o -name "*.h" | xargs clang-format -i

Create a .clang-format configuration file in your project root to customise rules:

# .clang-format
BasedOnStyle: Google
IndentWidth: 4
ColumnLimit: 120
BreakBeforeBraces: Stroustrup
PointerAlignment: Left

clang-tidy

clang-tidy is a linter and static analysis tool that goes beyond formatting. It detects bugs, performance issues, readability problems, and modernisation opportunities.

# Run clang-tidy with modernise checks
clang-tidy main.cpp -- -std=c++20

# Common checks
clang-tidy -checks="modernize-*,readability-*,bugprone-*" main.cpp

Create a .clang-tidy file for project-wide configuration:

# .clang-tidy
Checks: >
  -*,
  bugprone-*,
  modernize-*,
  readability-*,
  performance-*
WarningsAsErrors: ''
HeaderFilterRegex: '.*'

cppcheck

cppcheck is a standalone static analyser focused on detecting undefined behaviour, memory leaks, null-pointer dereferences, and other defects. It works without requiring a compilation database.

# Basic analysis
cppcheck --enable=all --std=c++20 src/

# Suppress specific warnings
cppcheck --suppress=missingIncludeSystem src/

Integrating Linters into Your Workflow

  • Editor integration – Most IDEs (VS Code, CLion, Visual Studio) support clang-format and clang-tidy natively or via extensions.
  • Pre-commit hooks – Run clang-format --dry-run --Werror before each Git commit to reject unformatted code.
  • CI/CD pipelines – Add linting steps to GitHub Actions, GitLab CI, or Jenkins to enforce style on every pull request.

Code Review Checklist

A structured checklist ensures that code reviews are thorough and consistent. Use the following categories when reviewing C++ code:

CategoryCheck
StyleDoes the code follow the project’s style guide? Has clang-format been run?
NamingAre names descriptive and consistent with conventions?
CommentsAre comments meaningful? Is the public API documented?
CorrectnessDoes the code do what it claims? Are edge cases handled?
SafetyIs memory managed correctly? Are raw pointers avoided where smart pointers apply?
PerformanceAre unnecessary copies avoided? Are containers used appropriately?
Error handlingAre errors checked and reported? Are exceptions used consistently?
TestingAre new features accompanied by unit tests?
DependenciesAre includes minimal? Are forward declarations used where possible?
SimplicityCould the code be simpler? Is there any dead code?

Key Takeaways

  • Consistency is more important than any particular style – pick a guide and stick to it.
  • Use descriptive names: PascalCase for types, snake_case or camelCase for variables/functions, ALL_CAPS for constants.
  • Choose a brace style (K&R, Allman, or Stroustrup) and apply it uniformly.
  • Keep lines under 80–120 characters and use blank lines to separate logical sections.
  • Comment why, not what. Use Doxygen for public API documentation.
  • Protect headers with #pragma once or traditional include guards.
  • Separate declarations (headers) from definitions (source files) and use forward declarations to reduce coupling.
  • Adopt a well-known style guide: Google, LLVM, Mozilla, or the C++ Core Guidelines.
  • Automate formatting with clang-format and catch bugs with clang-tidy and cppcheck.
  • Use a code review checklist to ensure quality across your team.
